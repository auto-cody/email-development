Here is a resource of methods used to combat unwanted email client renderings. This is not intended to be a collection of all email buildnig tricks.
Forcing Address/Phone Number Styling
The Problem
Mobile devices/clients do a good job of identifying addresses and phone numbers for their users to interact with. They do such a good job you should be using specific code just to combat it.
The Solution
Add the following code inside your header CSS style tag:

* [href]:not([style]) {
color: inherit !important;
text-decoration: none !important;
}
a[x-apple-data-detectors] {
color: inherit !important;
text-decoration: none !important;
font-size: inherit !important;
font-family: inherit !important;
font-weight: inherit !important;
line-height: inherit !important;
}

When inline styling fails, this code will overwrite the style that email clients/devices try to force. It should not have any impact on anything else. You may need to change color value from inherit to whatever text color you use.
Outlook Vertical Image Stretch
The Problem
Some Outlook platforms will stretch your image in the Y axis if you are using 2x or larger image size(retina). This will be very noticeable almost all of the time, but occasionally it can be mised if you are not paying close attention to renderings.
The Solution
Using HTML attributes to set the height and width will prevent this from happening. You can also use the inline CSS properties height:auto; and max-height simultaneously to control mobile rendering as long as the width is the same percent as desktop.

Sample Code:
 <img alt="This is a sample" border="0" style="display:block;width:100%;max-width:600px;height:auto;max-height:250px;" width="600" height="250" src="imagesource.png" />
Outlook Table Width Sretch
The Problem
The width of your tables will blow up and stretch to the full width of the Outlook email viewing window. This happens when a table is not constrained by a width set in pixels. Outlook will completely ignore percentage widths then assume the table should take up all available space.
The Solution
Setting pixel-based widths with the HTML width attribute will prevent the table stretch from happening. Also including inline CSS percentage based widths will allow the table to adjust for smaller screens. Outlook will only render with the HTML width and other platforms will prioritize the CSS widths, allowing a single short bit of code to pull double-duty for desktop and mobile without the use of mobile classes.

Sample Code:
<table border="0" cellspacing="0" cellpadding="0" style="width:100%;max-width:600px;" width="600">

Nested tables may not need set HTML pixel widths if the parent table is set up properly, but don't ignore nested table widths just because you can.
Dark Mode Colors
The Problem
Email clients and devices shift coded colors (backgrounds, borders, text) to be a darker scheme. White becomes black, black becomes grey, etc. This can cause some visual inconcistencies that negatively impact design or branding.
The Solution
This is the most complicated subject of email rendering and it would take a whole course to go over the different methods and solutions. The long story short is that success generally takes a creative team that is knowledgable about how to design emails with dark mode in mind. At the very least you should be working with PNG images that have transparent backgrounds. This is because any images that contain background colors will cause a visual break because they will not change colors with the rest of the coded background after the dark mode shift.

Some content colors can be forced via code, but not consistently across all platforms - at least not without hours of building and testing to find out if it is doable for your email design.
